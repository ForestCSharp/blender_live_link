Compute Tessellation Explained

The renderer can optionally expand Blender live-link triangle meshes into denser GPU vertex and index buffers before geometry-dependent passes run. The generated buffers are then reused by main geometry, shadows, GI capture, SSAO inputs, and the shaded wireframe visualization.

Executive Summary

Tessellation is a frame-local GPU geometry generation path. All tessellation modes now plan patch records in compute. Fixed mode computes exact draw counts on the CPU from source triangle count and factor, while adaptive modes read back a tiny counter buffer for GPU-chosen counts. Draw passes ask each mesh for a MeshRenderView, so the same downstream rendering code can use either source geometry or tessellated geometry. Skinned meshes use a GPU pre-skin cache so tessellation and shaded wireframe consume the same plain vertex layout as static meshes.

Planning

computeper frame

Choose patch factors, reserve generated ranges with atomics, apply budgets, and write compact patch metadata.

Emission

computestorage buffers

Generate interpolated vertices, Phong-smoothed positions, and triangle indices. Wire line indices are retained for legacy/debug paths.

Rendering

shared viewfallback

All geometry-dependent passes draw through mesh_get_render_view() and fall back to source mesh buffers.

Where It Runs In The Frame

The update happens after live-link object data and transform buffers are current, but before shadow, geometry, and lighting capture passes render geometry. 

object_copy_physics_transform(...)
object_update_storage_buffer(...)

GpuSkinning::update(state, tessellation_or_wireframe_needs_cache)
Tessellation::update(state, camera, fov)

ShadowDepthPass::render(...)
LightingCapture::render(...)
Geometry pass
SSAO reads geometry pass position/normal outputs

This ordering is the main reason tessellation shows up everywhere. Shadow depth and GI capture do not need their own tessellation logic; they simply receive the mesh view that points at the generated buffers.

Fallback rule: when tessellation is disabled, over budget, or has no generated indices, mesh_get_render_view() returns the original live-link vertex and index buffers.

Patch Planning

The renderer treats each source triangle as one or more tessellation patches, and plan_patches now writes those patch records for every mode. Fixed mode uses a uniform factor supplied by the CPU. Per-mesh adaptive mode runs measure_mesh_factor first, then writes one patch per source triangle using that mesh-wide factor. Per-triangle adaptive mode chooses per-edge LODs, optional virtual patch splits, and per-patch factors directly in plan_patches. 

vertices_per_patch = ((factor + 1) * (factor + 2)) / 2;
indices_per_patch = factor * factor * 3;

generated_vertex_count += vertices_per_patch;
generated_index_count += indices_per_patch;
generated_wire_index_count = generated_index_count * 2;

All modes use the GPU slot buffers. Fixed mode sizes them from exact counts and can activate the generated slot immediately after dispatch. Adaptive modes size them from worst-case budgets; rendering uses the most recent completed readback slot while the next slot is planned and emitted.

In per-triangle mode, large source triangles split into virtual patches inside the planner shader before factor selection. A virtual patch stores a smaller barycentric domain inside the original source triangle, so emission can still read the same three source vertices while generating denser local geometry.

Skinned source meshes are first converted into a mesh-local Vertex cache by a compute pass. The tessellation planner and shaded wireframe overlay read that cache instead of the bind-pose vertex buffer.

Tessellation Factors

Mode How factor is chosen Best use
Fixed Uses fixed_factor, clamped to 1..max_factor. Deterministic inspection, debugging counts, and comparing factor levels.
AdaptiveAngularPerMesh Measures world-space edge angular size relative to the active camera, converts that to approximate pixels, and gives every patch in the mesh the highest factor needed by any mesh edge. Keeping the legacy one-factor-per-mesh behavior for comparison and debugging.
AdaptiveAngularPerTriangle Measures world-space edge angular size relative to the active camera, converts that to approximate pixels, stores fractional per-edge LODs, optionally splits large triangles into virtual patches, and gives each patch the ceiling of its highest edge LOD. Keeping nearby or large-on-screen triangles denser without over-tessellating the whole mesh.

Per-triangle mode is the default selected mode. It also stores three edge LODs per patch so generated boundary vertices can morph toward the lower-density edge grid instead of leaving obvious T-junctions when neighboring triangles choose different factors.

Shared Buffer Layouts

Tessellation buffer structs are shared between C++ and GLSL through tessellation_common.h. C++ uses the same TessellationPatch and TessellationCounters declarations as the compute shader, with static assertions near mesh.h to catch layout drift.

static_assert(sizeof(TessellationPatch) == 80);

Compute Shader Flow

tessellation.glsl contains the compute entry points that plan and emit generated geometry. The CPU no longer uploads patch records.

skin_vertices

Runs before tessellation when a skinned mesh needs tessellation or shaded wireframe. It reads bind-pose vertices, joint weights, and current skin matrices, then writes a deformed mesh-local Vertex cache.

measure_mesh_factor

Runs only for Adaptive Angular (Per Mesh). It dispatches over source triangles, computes the maximum angular edge factor seen by the mesh, and stores that value in TessellationCounters.max_factor_seen.

plan_patches

Dispatches over source triangles for every mode. It writes TessellationPatch records, atomically reserves generated vertex/index ranges, and marks overflow when patch or generated buffer budgets are exceeded. Active adaptive skinned slots re-run emission against the current pre-skinned cache while waiting for newer readback counts, so animation poses stay current even when topology/counts are one readback behind.

emit_vertices_gpu

Builds each patch's triangular grid. It converts a local vertex index to barycentric coordinates, interpolates boundary coordinates by edge LOD, remaps through the patch barycentric domain, then interpolates source position, normal, and UV before optional Phong tessellation.

emit_indices_gpu

Walks the triangular grid and emits factor * factor small triangles. It also emits three line segments per generated triangle for legacy/debug wireframe data. 

for each skinned mesh that needs generated/debug geometry:
    skin_vertices(source vertices, skin weights, skin matrices -> skinned vertex cache)

for each tessellated mesh:
    clear_counters()
    if mode == Adaptive Angular (Per Mesh):
        measure_mesh_factor(source vertices, source indices -> counters.max_factor_seen)
    plan_patches(source vertices, source indices, counters -> patches)
    emit_vertices_gpu(source vertices, source indices, patches -> generated vertices)
    emit_indices_gpu(patches -> generated triangle indices, generated wire indices)

Phong Tessellation

Linear tessellation places new vertices directly on the original triangle plane. Phong tessellation bends those generated positions toward planes defined by the source vertex normals, which gives smoother silhouettes and highlights on curved meshes. 

linear_position = p0 * b0 + p1 * b1 + p2 * b2;

projected_position =
    project_to_normal_plane(linear_position, p0, n0) * b0 +
    project_to_normal_plane(linear_position, p1, n1) * b1 +
    project_to_normal_plane(linear_position, p2, n2) * b2;

out_position = mix(linear_position, projected_position, phong_strength);

phong_strength = 0 keeps the generated mesh on the original triangles. phong_strength = 1 uses the fully projected position.

Render Pass Integration

Geometry consumers do not inspect tessellation settings directly. They call mesh_get_render_view(), bind the returned buffers, and draw the returned counts. 

MeshRenderView render_view = mesh_get_render_view(mesh);

bindings.vertex_buffers[0] = render_view.vertex_buffer;
bindings.index_buffer = render_view.index_buffer;
sg_draw(0, render_view.index_count, 1);
Pass Uses tessellated geometry? Why it matters
Main geometry pass Yes Writes tessellated world position and normal into the G-buffer.
SSAO Indirectly Reads the G-buffer generated by the main geometry pass.
Shadow depth pass Yes Shadow maps see the same generated silhouettes and surface changes.
GI lighting capture Yes Probe cubemaps capture tessellated scene geometry.
Physics No Simulation continues to use the source mesh and transforms.

Shaded Wireframe

The visualization mode now runs after lighting and depth of field, before tonemapping. It copies the lit HDR image, draws the active mesh triangles again, and computes wire coverage analytically from per-corner barycentric values in the fragment shader. 

if (wireframe.shaded_wireframe)
{
    copy_lit_color_to_wire_overlay();
    sg_apply_pipeline(WireOverlayPass::get_mesh_overlay_pipeline(...));
    bindings.views[0] = render_view.vertex_storage_view;
    bindings.views[1] = render_view.index_storage_view;
    sg_draw(0, render_view.index_count, 1);
}

The overlay samples the geometry position buffer for visibility rejection, so hidden triangles do not draw over foreground geometry. It no longer writes fake wire material into the G-buffer or depends on hardware line rasterization.

Debug Stats

The UI reports mesh count, overflow count, source triangle count, patch count, generated vertex/index counts, maximum factor seen, readback backend support, and readback age. These numbers are reset and rebuilt every Tessellation::update() from fixed-mode exact counts or the latest completed adaptive GPU counter readback.

Current Limits

No Displacement

Tessellation currently smooths existing geometry. It does not sample displacement textures or change material schemas.

Triangle Patches

Live-link meshes arrive triangulated, so tessellation still starts from triangles. Virtual patches split large triangles in barycentric space, but quad-aware topology is deferred until source polygon metadata exists.

Readback Counts

Sokol draw calls still use CPU-visible counts. Adaptive modes select factors and write patch records in compute, then read back a tiny TessellationCounters buffer on Metal and D3D11. Fixed mode avoids readback because its counts are exact from source_triangle_count and fixed_factor. Unsupported backends warn and fall back for adaptive modes until their readback paths or indirect draw are implemented.

Edge Morphing Only

Per-triangle mode reduces visible boundary cracks with edge LOD morphing, but it is not an indirect draw path, occlusion culling, layered displacement, or mesh-shader compression path.

Practical debug path: start with Fixed mode at factor 1 to confirm source parity, switch to factors 2, 4, or 8 to inspect counts, then enable Shaded Wireframe and Phong Strength to see where smoothing changes the generated mesh.

From One Triangle To Tessellated Geometry

This walkthrough follows one source triangle through the exact CPU and GPU handoff. A source triangle starts as three indices in the mesh index buffer. Tessellation turns that triangle into one or more patch records, emits a triangular grid of generated vertices for each patch, emits triangle indices for that grid, then lets normal render passes draw the generated buffers through MeshRenderView. 

source index buffer:
    tri_index = N
    source_index_offset = N * 3
    i0 = indices[source_index_offset + 0]
    i1 = indices[source_index_offset + 1]
    i2 = indices[source_index_offset + 2]

source vertex buffer:
    v0 = vertices[i0]
    v1 = vertices[i1]
    v2 = vertices[i2]

Sequential Frame Flow

Step CPU work GPU work Output
1. Frame prep Object transforms, material buffers, camera matrices, and live-link mesh buffers are current before Tessellation::update() runs. None. Source Vertex and u32 index buffers are ready.
2. Eligibility If tessellation is disabled, the mesh has no triangles, the needed skinned cache is unavailable, or the last GPU-planned result overflowed, the mesh stays on the source buffers. None. Either fallback source rendering, or a mesh eligible for patch planning.
3. Slot setup The CPU computes source_triangle_count = index_count / 3, chooses an in-flight GPU slot, and sizes patch, vertex, index, and wire-index buffers. Fixed mode also computes exact generated counts from source_triangle_count and fixed_factor; adaptive modes allocate worst-case capacity from budgets and require readback support. clear_counters resets patch, generated-count, overflow, and max-factor counters. Empty GPU slot ready for planning.
4. Per-mesh measure Only Adaptive Angular (Per Mesh) dispatches this pass. Fixed and Adaptive Angular (Per Triangle) skip it. measure_mesh_factor dispatches over source triangles, reads i0/i1/i2, transforms positions to world space, computes angular edge LOD, and atomically updates counters.max_factor_seen. One mesh-wide adaptive factor stored in the counter buffer.
5. Patch planning The CPU dispatches plan_patches. It does not iterate source triangles in any tessellation mode. One workgroup handles each source triangle. Fixed mode writes one patch with uniform edge LODs from fixed_factor. Per-mesh mode writes one patch with uniform edge LODs from max_factor_seen. Per-triangle mode estimates per-edge LOD, chooses virtual splits when enabled, atomically reserves generated ranges, writes patch records, and sets overflow flags. GPU-owned patch buffer plus TessellationCounters.
6. Patch record The CPU never uploads patch records; it only supplies planner uniforms and buffer capacities. Each patch stores source triangle identity, generated vertex/index offsets, factor, per-edge LODs, counts, and a barycentric sub-domain. Full source triangles use corners (0,0), (1,0), (0,1); virtual patches store smaller domains. One compact description of how to tessellate a source-triangle region.
7. Vertex emission Dispatches emit_vertices_gpu across the slot's patch capacity. For each patch-local grid vertex, the shader converts local vertex index to triangular-grid barycentrics, morphs boundary coordinates by edge LOD, remaps through the patch sub-domain, interpolates source position, normal, and UV, applies optional Phong projection, then writes a generated Vertex. Generated vertex buffer filled at generated_vertex_offset + local_vertex_index.
8. Index emission Dispatches emit_indices_gpu across the slot's patch capacity. The shader walks each patch's triangular grid cells and emits factor * factor tiny triangles. Indices are patch-local plus the patch's generated vertex offset. Legacy/debug line indices are also emitted. Generated triangle index buffer filled at generated_index_offset.
9. Counts Fixed mode activates the slot immediately using its exact CPU-known counts. Adaptive modes request async readback of TessellationCounters; rendering uses the latest completed non-overflowed slot, or the source mesh if no valid slot is ready. Metal/D3D11 copy the counter buffer to CPU-readable staging memory for adaptive modes. CPU-visible draw counts for the generated slot.
10. Rendering Render passes call mesh_get_render_view(). If tessellation is active and valid, the view points at generated buffers; otherwise it points at source buffers. Geometry, shadow depth, GI capture, and related passes draw the returned buffers normally. Tessellated triangles appear anywhere normal mesh rendering is used.

What Happens Inside One Patch

A patch with factor n becomes a triangular grid with ((n + 1) * (n + 2)) / 2 vertices and n * n output triangles. The GPU never invents new source topology; every generated point is still derived from the original three source vertices and the patch's barycentric domain. 

for local_vertex_index in patch.vertex_count:
    row, col = triangular_grid_position(local_vertex_index)
    local_bary = barycentric_from_grid(row, col, patch.factor)

    if local_bary is on a patch edge:
        local_bary = morph_edge_coordinate(local_bary, patch.edge_lod_for_that_edge)

    source_uv = remap_local_bary_through_patch_domain(local_bary)
    source_bary = (1 - source_uv.x - source_uv.y, source_uv.x, source_uv.y)

    linear_position = p0 * source_bary.x + p1 * source_bary.y + p2 * source_bary.z
    projected_position = phong_project(linear_position, p0, n0) * source_bary.x
                     + phong_project(linear_position, p1, n1) * source_bary.y
                     + phong_project(linear_position, p2, n2) * source_bary.z

    out.position = mix(linear_position, projected_position, phong_strength)
    out.normal = normalize(n0 * source_bary.x + n1 * source_bary.y + n2 * source_bary.z)
    out.texcoord = uv0 * source_bary.x + uv1 * source_bary.y + uv2 * source_bary.z

Virtual Patch Example

For a huge source triangle, per-triangle GPU planning may decide that one factor-31 patch is not dense enough. Instead of raising the per-patch max factor, it splits the triangle into a power-of-two grid of virtual patches. Each virtual patch stores three barycentric domain corners inside the original triangle. 

source triangle domain:
    A = (0, 0)
    B = (1, 0)
    C = (0, 1)

depth 0:
    1 virtual patch covers A-B-C

depth 2:
    split_segments = 4
    up to 16 virtual patches cover smaller domains inside A-B-C

each virtual patch:
    reads the same source i0/i1/i2
    computes its own edge LODs and factor
    reserves its own generated vertex/index ranges
    emits its own local triangular grid
Key mental model: GPU planning decides how many patches exist and where their generated ranges live. GPU emission decides the actual generated vertices and triangle indices. Rendering does not know which path produced the buffers; it only draws the active MeshRenderView.
