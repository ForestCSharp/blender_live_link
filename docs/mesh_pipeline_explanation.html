Mesh Pipeline Explainer

This page follows mesh data from import-time CPU ownership through the per-frame GPU render views. The static and skinned paths share the same render-view abstraction, but skinned meshes can take either a direct skinning draw path or a pre-skinned cache path when tessellation or shaded wireframe needs plain vertices.

Static Meshes

Static meshes are the baseline path. Their source vertex and triangle index buffers are uploaded once, then each frame chooses whether passes should render those source buffers or a tessellated GPU output slot. There is no per-triangle CPU planning in the tessellation path.

CPU-Owned Mesh Data

make_mesh() builds persistent mesh state from imported mesh data: vertices, indices, wire indices, meshlet metadata, material slots, and per-object references.

vertices indices wire indices materials

GPU Source Buffers

The source Mesh::vertex_buffer and Mesh::index_buffer are both renderable and storage-readable, so raster passes can bind them traditionally while compute and overlays can fetch them.

vertex buffer index buffer storage views

Render View

mesh_get_render_view() is the handoff point. It returns source buffers when tessellation is inactive, or generated tessellation buffers when an active slot is valid.

source view tessellated view fallback

Sequential Frame Flow

Object transforms are updated.

Physics and scene changes write the object's current transform. object_update_storage_buffer() uploads object data used by later passes.

Optional tessellation plans patches on the GPU.

If tessellation is enabled, Tessellation::update() prepares GPU buffers, dispatches planning compute, emits tessellated vertices and indices, and selects the newest valid output slot. Fixed mode has CPU-known counts; adaptive modes use readback for temporary CPU-visible draw counts.

Passes ask the mesh for a render view.

Geometry, shadows, GI capture, and wire overlay all use MeshRenderView. This keeps pass code from caring whether the mesh is still source topology or generated tessellation output.

Raster passes bind the active buffers.

mesh_apply_render_bindings() binds the active vertex and index buffers. Static meshes use the non-skinned pipelines because positions and normals are already in mesh-local space.

Post-lighting wireframe fetches active triangle buffers.

The shaded wireframe overlay draws triangles after lighting and DOF. It fetches positions and indices from storage views and computes anti-aliased edge coverage from barycentrics in the fragment shader.

Static Data Flow

MeshInitData
  -> make_mesh()
  -> Mesh::vertex_buffer / Mesh::index_buffer
  -> optional Tessellation::update()
       -> GPU patch planning
       -> generated vertex/index buffers
  -> mesh_get_render_view()
  -> geometry, shadow, GI, wire overlay
Source rendering Uses Mesh::vertex_buffer and Mesh::index_buffer directly. This is the fallback path whenever tessellation is disabled, unsupported, overflowing, or has no completed adaptive slot yet.
Fixed tessellation CPU computes total patch and generated index counts from triangle count plus the fixed factor, then the GPU writes the generated buffers. The slot can be activated immediately because counts are exact.
Adaptive tessellation GPU compute measures and plans patch density. The CPU does not iterate source triangles; it consumes a small counter readback on supported backends to know the completed draw sizes.
Wire overlay Uses the same active render view as shaded rendering, but through storage views because the overlay shader fetches vertices and indices manually with gl_VertexIndex.
Static meshes do not need a deformation cache. Their source vertex buffer is already the correct mesh-local input for raster, compute tessellation, and the post-lighting wireframe overlay.

Skinned Meshes

Skinned meshes carry bind-pose vertices plus joint indices and weights. The default animated rendering path skins in the draw shader. Tessellation and shaded wireframe use a separate GPU pre-skinning cache so they can consume the same plain Vertex layout as static meshes.

Animation Inputs

Each skinned mesh has bind-pose Vertex data, SkinnedVertex joint data, a skin matrix buffer, and an armature mapping that converts animation pose into mesh-local deformation.

bind pose joint indices weights skin matrices

Direct Skinned Draw

With tessellation off and shaded wireframe off, passes can render the source buffers using skinned shader variants. This preserves the existing animation path and skin debug behavior.

skinned pipeline source topology no cache cost

Pre-Skinned Cache

When tessellation or shaded wireframe needs animated vertices, GpuSkinning::update() dispatches skin_vertices and writes deformed mesh-local vertices into skinned_vertex_cache_buffer.

plain Vertex output mesh-local compute generated

Sequential Frame Flow

Animation matrices are updated.

update_skinned_animations() advances the animation pose, computes each mesh's current skin matrices, and uploads them to the mesh skin matrix buffer.

Object storage is updated normally.

The object transform still applies after skinning. Skinning writes mesh-local deformed vertices, so world transforms, camera transforms, shadows, and GI continue to use the same object data model.

The optional pre-skin cache is refreshed.

GpuSkinning::update() runs only when tessellation is enabled or shaded wireframe is enabled. If neither feature needs plain animated vertices, cache validity is cleared and static meshes pay no cost.

The renderer chooses one of two skinned paths.

Without tessellation or wire overlay needs, passes render through skinned shader variants. With tessellation active, tessellation reads the pre-skinned cache and emits generated geometry that is already deformed.

Wire overlay consumes the right animated source.

If tessellation is active, shaded wireframe draws generated tessellation buffers. If tessellation is inactive, it draws the pre-skinned cache with the source index buffer so wires follow the animated pose.

Tessellation or Wireframe

bind-pose Vertex buffer
+ SkinnedVertex weights
+ skin matrix buffer
  -> gpu_skinning skin_vertices
  -> skinned_vertex_cache_buffer
  -> tessellation or wire overlay
  -> non-skinned generated render view
Tessellation off, wire off Uses the existing skinned draw path. The vertex shader reads bind-pose vertices, joint weights, and skin matrices, then skins positions and normals during rendering.
Tessellation off, wire on The pre-skin cache is generated for the current frame. The shaded wireframe overlay fetches deformed Vertex data from that cache and source triangle indices from the original index buffer.
Tessellation on Tessellation planning and emission read the pre-skinned cache as their source vertex buffer. Generated tessellated vertices are already animated, so geometry, shadows, GI, and wire overlay use non-skinned render views.
Adaptive readback latency Adaptive tessellation still uses completed counter readbacks for draw sizes. For active skinned slots, emission is refreshed from the current pre-skinned cache so the visible pose does not wait on the readback.

Why The Cache Exists

Tessellation planning, tessellation emission, and wire overlay all expect a plain Vertex storage buffer. Putting skinning into each of those shaders would duplicate animation logic across several compute and raster programs. The pre-skin cache centralizes deformation once per frame, then all downstream paths consume animated meshes as if they were static.

The cache is deliberately scoped: it does not rewrite animated bounds, it does not add CPU skinning, and tessellated generated geometry does not carry the existing skin debug color path in this first pass.
