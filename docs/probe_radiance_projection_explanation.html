probe_radiance_projection.glsl Explained

This compute shader projects one captured probe lighting cubemap into a compact coefficient representation. It writes either SH9 coefficients or SG9 lobe amplitudes, which the lighting shader can later evaluate instead of sampling the octahedral irradiance atlas.

Input Compute Output

The shader runs after a GI probe has rendered its six-face lighting cubemap. Each dispatch processes one probe. The workgroup has 32 lanes, but only lanes 0..8 do useful work: one lane per coefficient. 

layout(local_size_x=32, local_size_y=1, local_size_z=1) in;

uint coefficient_index = gl_GlobalInvocationID.x;
if (coefficient_index >= 9u)
{
    return;
}

The 32-wide workgroup is mostly a GPU/backend requirement. Sokol's shader compiler requires the compute workgroup size to be a multiple of 32 for this target, so the shader launches 32 threads and discards the extras.

Shared Probe Radiance Helpers

The shader includes shader_common.h and then imports the shared probe_radiance block. That block defines the coefficient struct and the math for SH and SG basis evaluation. 

#include "shader_common.h"

@cs cs

@include_block probe_radiance

The important shared type is intentionally tiny: one vec4 per coefficient. RGB stores radiance data; alpha is currently unused. 

struct ProbeRadianceCoefficient
{
    vec4 value;
};

Why the buffers are flat: coefficients are addressed as probe_index * 9 + coefficient_index. This keeps GPU storage simple and lets the lighting shader use the same indexing convention.

Sampling The Sphere

Both projection paths need a repeatable set of directions over the sphere. The shader uses a Hammersley sequence: one dimension is linear, the other uses the Van der Corput radical inverse. This gives a low-discrepancy pattern that converges more evenly than pure random sampling. 

float radical_inverse_vdc(uint bits)
{
    bits = (bits << 16u) | (bits >> 16u);
    bits = ((bits & 0x55555555u) << 1u) | ((bits & 0xAAAAAAAAu) >> 1u);
    bits = ((bits & 0x33333333u) << 2u) | ((bits & 0xCCCCCCCCu) >> 2u);
    bits = ((bits & 0x0F0F0F0Fu) << 4u) | ((bits & 0xF0F0F0F0u) >> 4u);
    bits = ((bits & 0x00FF00FFu) << 8u) | ((bits & 0xFF00FF00u) >> 8u);
    return float(bits) * 2.3283064365386963e-10;
}

vec2 hammersley(uint i, uint N)
{
    return vec2(float(i) / float(N), radical_inverse_vdc(i));
}

The 2D Hammersley point is mapped to a direction on the unit sphere. The mapping uses z = 1 - 2u.x, which covers the full range from the north pole to the south pole. 

vec3 sample_sphere(vec2 u)
{
    float z = 1.0 - 2.0 * u.x;
    float r = sqrt(max(0.0, 1.0 - z * z));
    float phi = 2.0 * M_PI * u.y;
    return vec3(r * cos(phi), r * sin(phi), z);
}

Main Dispatch Flow

Each active thread computes one coefficient for the selected probe. The offset into the coefficient buffer is: 

int coefficient_offset = probe_index * 9 + int(coefficient_index);
uint num_samples = uint(max(sample_count, 1));

This means dispatching once for probe_index = 17 writes coefficients 153..161 in the corresponding coefficient buffer.

SH9 Projection

In SH9 mode, each thread projects the cubemap radiance onto one real spherical-harmonic basis function. For coefficient k, it repeatedly samples a sphere direction dir, reads the cubemap radiance in that direction, multiplies by sh9_basis(k, dir), and accumulates. 

if (radiance_mode == PROBE_RADIANCE_MODE_SH9)
{
    vec3 accumulated = vec3(0.0);
    for (uint i = 0u; i < num_samples; ++i)
    {
        vec3 dir = sample_sphere(hammersley(i, num_samples));
        vec3 radiance = texture(samplerCube(cubemap_lighting_texture, smp), dir).rgb;
        accumulated += radiance * sh9_basis(int(coefficient_index), dir);
    }

    float integral_weight = (4.0 * M_PI) / float(num_samples);
    sh9_coefficients[coefficient_offset].value = vec4(accumulated * integral_weight, 0.0);
}

The 4π / sample_count factor converts the average sample sum into an approximation of an integral over the sphere. Later, the lighting shader evaluates those coefficients with diffuse convolution factors.

What SH9 Stores

SH9 is a low-frequency representation. It is good for smooth irradiance and broad directional color changes, but it cannot represent sharp lighting details.

Each probe gets nine RGB coefficients: one constant term, three first-order directional terms, and five second-order terms.

SG9 Projection

In SG9 mode, each coefficient corresponds to a fixed spherical-Gaussian lobe. The shader estimates the lobe's color amplitude by weighted averaging: samples near the lobe axis contribute strongly, samples far away contribute weakly. 

else if (radiance_mode == PROBE_RADIANCE_MODE_SG9)
{
    vec3 accumulated = vec3(0.0);
    float accumulated_weight = 0.0;
    for (uint i = 0u; i < num_samples; ++i)
    {
        vec3 dir = sample_sphere(hammersley(i, num_samples));
        vec3 radiance = texture(samplerCube(cubemap_lighting_texture, smp), dir).rgb;
        float weight = sg9_basis(int(coefficient_index), dir);
        accumulated += radiance * weight;
        accumulated_weight += weight;
    }

    sg9_coefficients[coefficient_offset].value =
        vec4(accumulated / max(accumulated_weight, 0.00001), 0.0);
}

The denominator normalizes the weighted average. The max(..., 0.00001) guard prevents division by zero, though the current fixed SG basis should always accumulate some positive weight.

What SG9 Stores

SG9 stores nine lobe amplitudes. The lobe axes and sharpness are shared constants in the probe_radiance block, not per-probe data.

This first pass uses a simple normalized weighted projection rather than a full least-squares fit, so it is intentionally inexpensive and easy to reason about.

What This Shader Does Not Do

No Visibility

Probe visibility remains in the octahedral depth/visibility atlas. Chebyshev and EVRP are handled elsewhere.

No Octahedral Radiance

The octahedral irradiance atlas path is still generated by cubemap_to_octahedral.glsl. This shader only writes SH9/SG9 coefficient buffers.

No Runtime Shading

Final surface GI is evaluated in lighting.glsl, which reads the coefficients written here.

End-To-End Data Flow

LightingCapture::render(...)
    - render probe geometry cubemap
    - render probe lighting cubemap
    - render radial visibility cubemap
    - update octahedral visibility atlas
    - if mode is SH9 or SG9:
        dispatch probe_radiance_projection.glsl
        write 9 coefficients for this probe

lighting.glsl
    - Octahedral mode: sample octahedral irradiance atlas
    - SH9 mode: evaluate SH9 coefficient buffer
    - SG9 mode: evaluate SG9 coefficient buffer
    - all modes: use octahedral visibility atlas for probe occlusion
